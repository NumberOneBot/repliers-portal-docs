# Map Overlays

> How the search map draws **overlay layers** — schools, neighborhoods, postal
> codes, school districts — on top of the listings, and how selection/popups work.
> For the broader map/search anatomy see [Property Search — How it works](/product-guide/search/technical).

Overlays are extra, config-driven layers of geographic data (from the **LiveBy**
source). Each overlay is a plain data object — an `OverlayLayerDefinition` — so a
tenant enables, reorders, restyles, or hides overlays entirely through config,
without touching the map code. The default set is defined in
`src/configs/defaults/overlays/liveBy.ts` and gated on `features.liveBy`.

## Key files

| File                                                    | Role                                                                                                                                             |
| ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| `src/configs/defaults/overlays/liveBy.ts`               | The overlay definitions (`livebyNeighborhood`, `livebyPostalCode`, `livebySchoolDistrict`, `livebySchool`) + the exported `overlayLayers` array. |
| `src/configs/defaults/map.ts`                           | `OverlayLayerDefinition` type + the `overlays: { layers, persistSelection }` config block.                                                       |

| `.../hooks/useOverlayLayers/useOverlayLayers.ts`        | The per-overlay sync engine: fetch → render polygons (GL) + markers (DOM).                                                                       |
| `.../hooks/useOverlayLayers/layers.ts` + `polygon.ts`   | Add/remove the GL **fill/line** polygon layers.                                                                                                  |
| `.../hooks/useOverlayMarkers/renderMarkers.ts`          | Render overlay point features as **DOM** markers + client-side clustering.                                                                       |
| `.../hooks/useOverlaySelectsLocation.ts`                | Wires marker/polygon clicks → location selection (search map only).                                                                              |
| `.../Map/markerElement.ts`                              | Builds the marker DOM element (shared by listing + overlay markers).                                                                             |
| `.../Map/OverlayMarkerPopup/`                           | Popup host + per-overlay tooltip cards (`tooltipRegistry.ts`).                                                                                   |

## GL vs DOM

Mapbox is used two ways on this map — know which is which:

- **Listing markers → DOM.** Search-result markers are plain DOM nodes built by
  `createMarkerElement` (`markerElement.ts`) and driven by `useListingMarkers`.
  Cheap: ~100 markers created in well under a frame, no per-marker React root.
- **Overlay point markers → DOM (current reality).** School, neighborhood, and
  postal-code point/cluster markers are **also DOM markers**, rendered by
  `renderOverlayMarkers` (`useOverlayMarkers/renderMarkers.ts`) via the same
  `createMarkerElement` path. They were previously GL circle layers — that path
  is gone. Clustering is done client-side in `cluster.ts` (not by a GL `cluster: true` source).
- **Overlay polygons → GL.** Boundaries and catchments render as GL **fill + line**
  layers (`addOverlayPolygons` → `addPolygonToMap`). Polygons must stay GL — DOM
  can't draw arbitrary geometry.

<Callout type="info">
  A single overlay commonly renders **both**: DOM markers *and* a GL polygon
  (e.g. a school marker with a `showOnMarkerHover` catchment polygon). Polygons
  render **data-driven** — whenever the fetched data contains polygon features —
  the `polygon` config only tunes *how* (colour/style/hover behaviour).
</Callout>

## How it works

1. **Definition** — each overlay in `overlayLayers` declares its `label`, `color`,
   `fetchData` (bounds → GeoJSON), `activationMinZoom`, and how it's `selectable`
   (`'marker'` | `'polygon'` | none).
2. **Toggle menu** — the search map's `MapLayersMenu` lists overlays where
   `showOn.search` is true (and `showOn.menu !== false`); toggling one flips
   `activeLayers` in `MapOptionsProvider`.
3. **Sync** — `useOverlayLayers` reacts to `activeLayers` + viewport changes: for
   each enabled layer it fetches data for the current bounds, adds the GL polygon
   layer, and renders DOM markers. Layers below their `activationMinZoom` are
   removed and re-fetched once zoomed back in.
4. **Selection → popup** — on the search map, `useOverlaySelectsLocation` turns a
   marker/polygon click into a location selection (`MapOptionsProvider.locations`
   - the `locationId` filter). Hovering shows a popup: `OverlayMarkerPopup`
     renders the React card looked up by overlay id in `tooltipRegistry`.

## Configuration

- **`@configs/overlays`** (`overlays/liveBy.ts`) — the overlay definitions. Add,
  reorder, restyle, or drop layers here; tenants override the file.
- **`@configs/map` → `overlays`** — `{ layers: overlayLayers, persistSelection }`.
  `layers` is the ordered list; `persistSelection: false` (default) drops a
  layer's selected locations when it's hidden.
- **Per-layer availability** is controlled by `showOn` (`search` opt-in, `listing`
  opt-out, `menu` to hide the row) — not a separate toggle-menu config object.
- **Tooltips** — register a card component per overlay id in
  `OverlayMarkerPopup/tooltipRegistry.ts` (override via `_<tenant>/tooltipRegistry.tsx`).

## Gotchas & hints

- **Mapbox serializes feature properties as JSON strings.** Properties read off a
  clicked/queried GL feature come back stringified — booleans become `'true'`,
  arrays/objects become JSON text. Parse non-scalars before use (see
  `LiveBySchoolCard.tsx`'s `asBool`, and parse array/object props with `JSON.parse`).
- **Polygons must stay GL.** Never try to move polygon geometry onto the DOM path —
  only points/clusters are DOM markers.
- **Overlays are gated by data + config presence.** Nothing renders if
  `features.liveBy` is off (empty `overlayLayers`), if the layer isn't in
  `activeLayers`, or if the fetch returns no matching features. Polygon existence is
  checked at runtime via `map.getSource(...)`, never by the `polygon` config field.
- **DOM markers survive `setStyle`; GL layers don't.** A map-style switch wipes GL
  sources/layers (polygons are re-added by the re-sync), while DOM markers persist
  and are recoloured in place to avoid flicker.
