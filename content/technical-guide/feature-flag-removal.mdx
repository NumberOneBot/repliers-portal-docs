# Feature Removal

When a [feature flag](/getting-started/capabilities-and-flags) is permanently OFF for a
tenant, the code behind it is still shipped — it just never renders. For a slimmed tenant
build (e.g. a standalone frontend export) you can physically delete that code. This page
maps, per flag, **what is actually deletable** versus what must stay because it is shared.

<Callout type="info">
  Deletable surface is very uneven across flags. "Big flag" does not mean "big
  deletion": route-gated features (`estimate`, `blog`) delete cleanly, while
  data-layer features (`locations`, `map`) look huge but their
  config/API/service spine is shared app-wide.
</Callout>

## How flags gate code

Each App Router page checks the flag and 404s when it is off:

```tsx
// app/<route>/page.tsx
if (!features.locations) return <Page40XTemplate />
```

The route still exists and the whole imported subtree stays in the bundle — so **deletable =
everything exclusively reachable from that feature**. Anything imported by another feature is
shared and must stay.

<Callout type="warning">
  **Utilities are mostly kept.** Helpers in `src/utils/` genuinely stitch
  features together (URL building, formatters, filters, listing visibility).
  Treat a util as deletable only when it is provably feature-exclusive.
</Callout>

## Account & personalization

Route pattern `app/(ClientPages)/<route>/page.tsx` + `components/pages/<page>/**`. The trap:
the **provider/hook** is usually mounted app-wide and driven by a button on listing cards or
the map, so it survives the page — unless that button is itself flag-gated.

- **`recentlyViewed`** — ~4 files. route + `components/pages/recently-viewed/**`. No provider/API → fully deletable.
- **`profile`** — ~9 files. route + `components/pages/profile/**` (incl. Unsubscribe, FeedbackForm). No provider/API → fully deletable.
- **`favorites`** — ~11 files. route + `components/pages/favorites/**` + `FavoriteRemoveDialog.tsx` + `FavoritesMenuItem.tsx` + `services/API/APIFavorites.ts`. Keep `FavoritesProvider`+`useFavorites` (listing-card favorite button).

Keep `ImageFavoritesProvider`+`useImageFavorites` (photo StarButton).

## Rules of thumb

- **Route-gated** features (`estimate`, `blog`, `dashboard`, `buildings`) delete cleanest — the page tree is self-contained.
- **Provider-backed** features (`favorites`, `saveSearch`, AI) delete the page but keep the app-wide provider unless its trigger (card/map button) is also flag-gated.
- **Data-layer** features (`locations`, `map`) look huge but the config/API/service spine is shared — only the pages/interaction delete.
- **Cross-tenant / CMS infra** (WordPress client, `ContactForm`, the overlay engine) is shared — verify per-tenant before a global deletion.
- **Watch the hidden gates:** `messaging` → OTP auth · `buildings` → MLS condo listings · `estimate` ↔ `dashboard` Statistics.
- After any removal, run `pnpm --filter portal-frontend type-check` for the tenant — a standalone tenant export is the first time a tenant's real code paths are fully type-checked.
