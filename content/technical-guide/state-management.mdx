# State Management

Portal uses **React Context only** (no Redux/Zustand). State is split between a small set of
**global** providers (mounted app-wide) and many **scoped** providers (mounted only inside
the section that needs them).

## Global providers

Composed in `src/app/_providers.tsx`, outermost → innermost:

```
NextIntlClientProvider        i18n messages / locale / timeZone
└─ AppRouterCacheProvider     MUI ↔ Next.js App Router cache
   └─ ThemeProvider           MUI theme (from styles/theme + @configs/theme)
      └─ UserProvider         auth state, profile, role, token refresh   → useUser()
         └─ DialogProvider    modal/dialog visibility & stacking          → useDialog(name)
            └─ SnackbarProvider  toast notifications                       → useSnackbar()
               └─ [FavoritesProvider?]      saved listings                → useFavorites()
                  └─ [SaveSearchProvider?]  saved searches                → useSaveSearch()
                     └─ [ImageFavoritesProvider?]  saved photos           → useImageFavorites()
```

<Callout type="info">
  The bottom three are **feature-gated at build time** — mounted only when
  `features.favorites` / `features.saveSearch` / `features.imageFavorites` are
  on for the tenant. Feature flags are static config, not a runtime provider
  (there is no `FeaturesProvider`).
</Callout>

These mount inside a specific route/template, close to where their state is used. Full
per-section detail is on each section's **How it works** page — see [Site Anatomy](/technical-guide/site-anatomy).

| ------------------------------------------------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ | ----------------------------------------------- |
| `SearchProvider`                                        | `useSearch`, `useSearchActions`                                                         | Filters, results, clusters, multi-units. Split read/write contexts.                        | Search, dashboard, locations, listing (similar) |

| `LiveByDemographicsProvider`                            | `useLiveByDemographics`                                                                 | LiveBy demographics for the listing location.                                              | `ListingPageTemplate`                           |
| `LocationProvider`                                      | `useLocationPage`, `useLocationMap`                                                     | Location metadata, tree, nearby, map focus.                                                | Locations & condos index                        |
| `BuildingProvider`                                      | `useBuilding`                                                                           | Single building/condo (CMS).                                                               | Building/condo detail                           |

## Consuming a provider

Always read through the provider's hook (never `useContext` directly), and read from the
narrowest hook that has what you need:

```tsx
import { useSearch, useSearchActions } from 'providers/SearchProvider'

// Reader: re-renders when results/filters change
const { listings, loading, filters } = useSearch()

// Writer: stable identity — does NOT re-render on results changes
const { setFilters } = useSearchActions()
```

<Callout type="info">
  Several providers split into multiple contexts to control re-renders.
  `SearchProvider` separates reads (`useSearch`) from stable writers
  (`useSearchActions`); `MapOptionsProvider` exposes five contexts (options,
  locations, layers, popup state, popup actions) so a marker popup update
  doesn't re-render the whole map tree.
</Callout>

## Provider structure

Multi-file providers follow this shape:

```
src/providers/SearchProvider/
├── SearchProvider.tsx     provider component (+ split contexts)
├── types.ts               context value types
├── mock.ts                default/mock context values
└── index.ts               exports (provider + hooks)
```

Simpler providers are a single file (e.g. `src/providers/UserProvider.tsx`).

## Best practices

1. **One domain per provider** — keep responsibilities narrow.
2. **Consume via hooks** — never `useContext` directly; read the narrowest hook.
3. **Split contexts for re-render control** — separate stable writers from changing reads.
4. **Scope by default** — mount a provider in the section that needs it, not app-wide, unless
   the state is genuinely global.
5. **Prefer config/self-sufficiency over prop plumbing** — read `@configs/*` at the point of
   use instead of threading values through props.
